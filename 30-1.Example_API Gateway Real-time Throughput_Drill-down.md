# API Gateway 실시간 처리량 + 에러 로그 드릴다운 모니터링 설계

"집계 지표만 보여주는 대시보드"와 "에러 났을 때 그 순간의 실제 로그 원문까지 드릴다운"은 설계 방식이 다릅니다. 이 문서는 이 요구사항에 맞는 데이터 모델과, Grafana/Metabase 비교를 정리합니다.

## 핵심 설계 원칙 — "집계 테이블 + 원본 로그 테이블"을 함께 둔다

드릴다운이 되려면 집계된 그래프의 특정 지점(시간대, 에러 그룹)을 클릭했을 때 **그 조건에 해당하는 원본 로그 행들을 다시 조회**할 수 있어야 합니다. 그러려면:

1. **원본 로그 테이블**에 구조화된 컬럼(필터링용) + **원문 로그 전체 텍스트**(사람이 읽을 최종 내용)를 함께 저장
2. **실시간 집계 테이블**은 그래프/처리량 표시용으로 짧은 주기(예: 10초~1분)로 미리 계산
3. 대시보드에서 "집계 그래프 → 원본 로그 테이블" 방향으로 필터(시간범위 + 에러조건)를 넘겨주는 **드릴다운 링크**를 건다

## 1. 원본 로그 테이블 — 드릴다운의 핵심

```sql
CREATE TABLE apigw_logs (
    timestamp        DateTime64(3),
    request_id       String,
    client_ip        String,
    method           LowCardinality(String),
    path             String,
    api_key          LowCardinality(String),
    status_code      UInt16,
    latency_ms       UInt32,
    upstream_service LowCardinality(String),
    response_size    UInt32,
    error_message    String,      -- 에러 상세 메시지(있는 경우)
    raw_log          String       -- 원본 로그 라인 전체(JSON 원문) - 드릴다운 시 그대로 보여줄 용도
) ENGINE = MergeTree()
PARTITION BY toYYYYMMDD(timestamp)
ORDER BY (upstream_service, status_code, timestamp)
TTL timestamp + INTERVAL 30 DAY;   -- 원본 로그는 집계보다 짧게 보관해도 무방 (용량 고려)
```

- `raw_log`에 수집기(Vector 등)에서 넘어온 **원본 JSON 문자열 그대로**를 함께 저장해두면, 드릴다운 화면에서 "이 요청의 실제 로그 원문"을 그대로 보여줄 수 있습니다.
- `ORDER BY`에 `status_code`를 넣어서 "에러(4xx/5xx)만 빠르게 필터링"하는 조회가 빨라지도록 했습니다.

## 2. 실시간 처리량 — 짧은 주기 집계

```sql
CREATE TABLE apigw_realtime_stats (
    ts_bucket        DateTime,          -- 10초 단위 버킷
    upstream_service LowCardinality(String),
    status_class     LowCardinality(String),  -- '2xx','4xx','5xx'
    request_count    AggregateFunction(count),
    avg_latency      AggregateFunction(avg, UInt32)
) ENGINE = AggregatingMergeTree()
ORDER BY (upstream_service, status_class, ts_bucket)
TTL ts_bucket + INTERVAL 7 DAY;   -- 실시간용은 짧게만 유지, 장기추세는 apigw_stats_1m 별도 사용

CREATE MATERIALIZED VIEW apigw_realtime_stats_mv TO apigw_realtime_stats AS
SELECT
    toStartOfInterval(timestamp, INTERVAL 10 SECOND) AS ts_bucket,
    upstream_service,
    multiIf(status_code < 300, '2xx', status_code < 500, '4xx', '5xx') AS status_class,
    countState() AS request_count,
    avgState(latency_ms) AS avg_latency
FROM apigw_logs
GROUP BY ts_bucket, upstream_service, status_class;
```

**대시보드에서 쓸 조회 (5~10초 자동 새로고침)**:

```sql
SELECT
    ts_bucket,
    upstream_service,
    status_class,
    countMerge(request_count) AS reqs,
    avgMerge(avg_latency)     AS avg_latency_ms
FROM apigw_realtime_stats
WHERE ts_bucket >= now() - INTERVAL 15 MINUTE
GROUP BY ts_bucket, upstream_service, status_class
ORDER BY ts_bucket;
```

## 3. 에러 드릴다운 쿼리 — 그래프에서 클릭 시 실행될 쿼리

```sql
-- 대시보드 변수로 $from, $to(클릭한 시간범위), $service를 받는다고 가정
SELECT
    timestamp,
    request_id,
    client_ip,
    method,
    path,
    status_code,
    latency_ms,
    error_message,
    raw_log
FROM apigw_logs
WHERE timestamp BETWEEN $from AND $to
  AND upstream_service = $service
  AND status_code >= 400
ORDER BY timestamp DESC
LIMIT 200;
```

이 결과 테이블에서 특정 `request_id` 행을 다시 클릭하면 `raw_log` 컬럼을 펼쳐서 **로그 원문 전체**를 보여주는 2차 드릴다운도 가능합니다.

## Grafana vs Metabase — 이 요구사항 기준 비교

| 요구사항 | Grafana | Metabase |
|---|---|---|
| 실시간(5~10초) 자동 갱신 | ◎ 기본 지원, 초 단위 refresh 설정 가능 | △ 자동 새로고침 있으나 보통 분 단위 권장, 초 단위는 부자연스러움 |
| 그래프 → 원본 로그 드릴다운 | ◎ Data Links 기능으로 그래프의 특정 포인트/바 클릭 시 시간범위+조건을 다른 패널/대시보드로 전달 | ○ Click behavior로 "커스텀 대상(다른 질문)"으로 이동하며 클릭한 값을 필터로 전달 가능 |
| 로그 원문(raw_log) 텍스트 뷰 | ◎ Logs 패널 타입이 원문 텍스트 표시에 최적화 | △ 테이블 형태로만 표시, 텍스트 전용 뷰는 약함 |
| SQL 자유도 (임의 파라미터 쿼리) | ◎ Raw SQL + 템플릿 변수 자유롭게 조합 | ○ SQL 질문 작성 가능하나 변수 바인딩이 Grafana보단 제한적 |
| 초기 설정 난이도 | 다소 높음 (변수/링크 설정 학습 필요) | 낮음 (질문 만들고 클릭 동작만 지정하면 됨) |
| 알림(에러율 급증 시 Slack 등) | ◎ Alerting 기능 강력 | ○ 가능하나 Grafana보단 단순 |

### 결론 — 이 요구사항이라면 Grafana 추천

"실시간 처리량 + 에러 발생 시 즉시 원본 로그 드릴다운"이라는 조합은 원래 Grafana가 인프라/로그 모니터링용으로 설계된 시나리오와 정확히 일치합니다. 특히:

- **Logs 패널**로 `raw_log` 컬럼을 그대로 시간순 스트림처럼 보여줄 수 있고
- **Data Links**로 처리량/에러율 그래프의 특정 시간대를 클릭하면 그 구간의 원본 로그만 필터링된 화면으로 바로 이동시킬 수 있습니다.

Metabase는 정기 리포트, 비즈니스 지표(주간 매출 추이 등) 중심의 BI 용도에는 더 쉽고 좋지만, **초 단위 실시간성 + 로그 원문 드릴다운**이 핵심이라면 Grafana 쪽이 훨씬 자연스럽습니다.

## 실무 대시보드 구성 제안 (Grafana 기준)

1. **상단 패널**: 처리량(req/s) 시계열 — `apigw_realtime_stats` 조회, 5~10초 refresh
2. **중단 패널**: 에러율(%) 시계열 — 2xx/4xx/5xx 스택 바 차트
3. **하단 패널**: 에러 로그 테이블 — 위 그래프에서 클릭한 시간대의 `status_code >= 400` 행들 (Data Link로 연결)
4. **팝업/서브패널**: 특정 `request_id` 클릭 시 `raw_log` 원문 텍스트 표시 (Logs 패널)
