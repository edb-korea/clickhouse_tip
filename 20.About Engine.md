# ClickHouse `ENGINE` 절 상세 설명

ClickHouse의 `CREATE TABLE` 문에서 `ENGINE = ...`는 **이 테이블이 데이터를 어떻게 저장하고, 조회하고, 병합/복제/삭제를 어떻게 처리할지 결정하는 핵심 절**입니다. 다른 RDBMS는 보통 스토리지 엔진이 하나(또는 몇 개로 고정)인데, ClickHouse는 약 28개의 테이블 엔진을 제공하며 테이블마다 다른 엔진을 선택할 수 있습니다.

## ENGINE이 결정하는 것

테이블 엔진(테이블의 유형)은 데이터가 어디에 어떻게 저장되는지, 어떤 서버에 데이터를 써야 하고 어디서 읽어야 하는지, 지원되는 쿼리 종류, 동시성 있는 데이터 접근이 가능한지, 인덱스가 존재하는지, 멀티스레드 요청 실행이 가능한지, 데이터 복제에 파라미터를 쓸 수 있는지 등을 결정합니다.

## 4가지 엔진 계열

ClickHouse는 크게 Log, MergeTree, Integration, Special 네 가지 계열로 나뉘며, 여기에 기능적으로 독립적인 두 특수 엔진(Replicated, Distributed)이 추가로 존재합니다.

---

### 1. MergeTree 계열 — 프로덕션 표준

MergeTree 계열은 ClickHouse 데이터 저장 능력의 핵심입니다. 컬럼형 저장, 커스텀 파티셔닝, 희소 기본 인덱스, 보조 data-skipping 인덱스 등 복원력과 고성능 데이터 조회를 위한 대부분의 기능을 제공합니다. 기본 MergeTree 테이블 엔진은 단일 노드 ClickHouse 인스턴스의 기본 엔진으로 볼 수 있는데, 다양한 사용 사례에 다재다능하고 실용적이기 때문입니다. 프로덕션 환경에서는 ReplicatedMergeTree를 쓰는 것이 정석인데, 일반 MergeTree의 모든 기능에 고가용성을 더해주기 때문입니다.

**동작 원리**: MergeTree는 데이터를 "파트(parts)"라는 불변의 정렬된 덩어리로 저장합니다. 각 INSERT마다 하나 이상의 새 파트가 생성되고, ClickHouse가 백그라운드에서 작은 파트들을 큰 파트로 병합해서 파트 개수를 적정 수준으로 유지합니다.

```sql
CREATE TABLE events (
    id UInt64,
    ts DateTime,
    user_id UInt32,
    region LowCardinality(String)
)
ENGINE = MergeTree()
PARTITION BY toYYYYMM(ts)
ORDER BY (region, ts);
```

- **`ORDER BY`**: 기본 정렬 키를 정의합니다. 각 파트 내 데이터가 이 키로 정렬되어 효율적인 범위 스캔을 가능하게 합니다. MergeTree 계열에서는 필수 절입니다.
- **`PARTITION BY`**: 데이터를 물리적으로 나누는 기준(보통 월별 등). 오래된 데이터를 파티션 단위로 저렴하게 삭제할 수 있게 해줍니다.
- **주의**: 기본키가 중복 방지(deduplication) 용도로 쓰이지 않습니다. 같은 기본키를 가진 여러 행을 삽입하는 것도 가능합니다. 즉 RDBMS의 PK와 달리 여기선 순수하게 "조회 가속용 정렬 인덱스"입니다.

#### MergeTree의 주요 변형(Variant)

| 변형 | 용도 | 핵심 동작 |
|---|---|---|
| **ReplacingMergeTree** | 중복 제거가 필요한 변경 가능 데이터 | 백그라운드 컴팩션 시 같은 기본키를 가진 중복 행을 제거합니다. 최종적(eventual) 중복 제거가 허용되는 경우에 사용하며, 쿼리 시점의 유일성을 보장하지는 않습니다. |
| **SummingMergeTree** | 단순 합계형 카운터 | 같은 기본키를 가진 행들을 숫자 컬럼 합산으로 하나의 행으로 사전 집계합니다. 저장공간을 줄이고 집계 쿼리 속도를 높여줍니다. 컴팩션 완료 전에는 일부 데이터가 아직 집계 안 된 상태일 수 있어, 쿼리에는 항상 SUM()과 GROUP BY를 함께 써야 합니다. |
| **AggregatingMergeTree** | 복잡한 사전계산 지표 | 평균, 백분위수, HyperLogLog 카운트 같은 복잡한 집계용. 대시보드처럼 미리 계산된 지표가 필요한 대규모 상황에 적합 |
| **CollapsingMergeTree** | 업데이트/삭제 흉내 | sign 컬럼을 사용해서 행을 상쇄시키는 방식으로 업데이트/삭제를 구현 |
| **VersionedCollapsingMergeTree** | 위와 유사 + 버전 컬럼으로 순서 보장 | 분산 삽입 순서가 꼬여도 정확히 상쇄되도록 버전 컬럼 추가 |

**ReplacingMergeTree 예시:**
```sql
CREATE TABLE products (
    product_id UInt64,
    name String,
    price Decimal64(2),
    stock_quantity UInt32,
    updated_at DateTime
) ENGINE = ReplacingMergeTree(updated_at)
ORDER BY product_id;
```

**SummingMergeTree 예시:**
```sql
CREATE TABLE test_tbl_summing (
    key UInt32,
    value UInt32
) ENGINE = SummingMergeTree()
ORDER BY key;
```

### 2. Log 계열 — 가벼운 소규모 테이블용

기능이 최소화된 가벼운 엔진들입니다. 약 100만 행 이하의 많은 작은 테이블을 빠르게 쓰고, 나중에 전체를 통째로 읽어야 할 때 가장 효과적입니다.

- `TinyLog`, `Log`, `StripeLog` 등이 여기 속합니다.
- 인덱스, 병합, 파티셔닝 같은 고급 기능이 없는 대신 구조가 단순해서 오버헤드가 적습니다.
- 테스트용 임시 테이블이나 아주 작은 참조 테이블에 적합합니다.

---

### 3. Integration 계열 — 외부 시스템 연동

다른 데이터 저장/처리 시스템과 통신하기 위한 엔진들입니다. ClickHouse를 Kafka, S3, PostgreSQL 같은 외부 시스템에 연결해주는 엔진입니다. 이 엔진들로는 ClickHouse에 데이터를 실제로 저장하지 않습니다. ClickHouse 인터페이스를 통해 외부 시스템을 조회하는 것입니다.

- **Kafka**: Kafka 토픽에서 직접 읽어와서, 메시지가 도착하는 대로 ClickHouse 테이블에 행을 삽입
- **S3**: Amazon S3나 호환 오브젝트 스토리지에 저장된 파일을 ClickHouse로 먼저 로드하지 않고도 조회
- **PostgreSQL / MySQL**: ClickHouse 쿼리 안에서 라이브 PostgreSQL/MySQL 테이블을 읽어와, 운영 데이터와 분석 데이터를 조인할 때 유용

앞서 다뤄드린 ClickHouse의 `PostgreSQL` 테이블 엔진이 바로 이 계열입니다.

---

### 4. Special 계열 — 특수 목적

Memory 엔진은 데이터를 메모리에 저장하며 재부팅 시 유실될 수 있습니다. 뛰어난 쿼리 성능을 제공하며, 데이터 영속성이 필요 없는 1억 행 미만의 작은 테이블에 적합합니다. 보통 ClickHouse에서 임시 테이블로 사용됩니다. Buffer 엔진은 대상 테이블에 메모리 버퍼를 설정하며, 버퍼가 특정 조건을 만족하면 디스크로 플러시됩니다.

- **Distributed**: 저장 엔진이 아닙니다. 단일 노드에서 클러스터로 옮겨갈 때 마주치게 되며, 단일 노드 배포에서는 필요 없습니다. 여러 샤드에 흩어진 MergeTree 테이블들을 하나의 논리적 테이블처럼 조회하게 해주는 라우팅 역할입니다.
- **Merge**: 여러 테이블을 통합 조회
- **Null**: 데이터를 저장하지 않고 버림 (파이프라인 테스트용)

---

## 실무 선택 가이드

결정은 엔진 개수만큼 복잡하지 않습니다: 반복 조회할 운영 데이터를 저장한다면 → MergeTree. 삽입 시 중복 제거가 필요하면 → ReplacingMergeTree. 사전집계된 카운터를 만든다면 → SummingMergeTree. 테스트용 작은 임시 테이블이면 → TinyLog나 Log. Kafka, S3, 외부 DB에서 읽어온다면 → 해당 Integration 엔진. 멀티노드 클러스터를 운영한다면 → MergeTree 테이블 위에 Distributed 테이블을 추가.

시작하는 단계라면 MergeTree를 선택하고, 가장 자주 쓰는 쿼리 필터를 기준으로 ORDER BY를 정의한 뒤 진행하세요. 엔진 유형은 나중에 테이블을 마이그레이션해서 바꿀 수 있지만, 처음부터 MergeTree를 제대로 설정하는 것이 대부분의 팀에게 필요한 것의 대부분을 커버합니다.

## 정리 — ENGINE 선택 흐름도

```
저장할 데이터가 자주 조회되는 운영/분석 데이터인가?
  → 예: MergeTree 계열 (기본 선택지)
      ├─ 중복 제거 필요? → ReplacingMergeTree
      ├─ 숫자 합산 집계? → SummingMergeTree
      ├─ 복잡한 집계(평균/백분위)? → AggregatingMergeTree
      └─ 업데이트/삭제 흉내? → CollapsingMergeTree

저장이 목적이 아니라 외부 시스템 조회가 목적인가?
  → Integration 계열 (Kafka, S3, PostgreSQL, MySQL 등)

아주 작고 일회성/테스트용 테이블인가?
  → Log 계열 (TinyLog, Log, StripeLog)

멀티노드 클러스터에서 샤드들을 통합 조회해야 하는가?
  → Special 계열의 Distributed (MergeTree 테이블 위에 얹음)
```
