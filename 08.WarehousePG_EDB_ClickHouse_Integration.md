# WarehousePG/EDB와 ClickHouse 연동

## 0. 구성 목적

이 문서는 WarehousePG/EDB(PostgreSQL 호환 DB)의 데이터를 ClickHouse에서 조회하거나 ClickHouse 테이블로 적재하는 방법을 정리합니다.

ClickHouse는 PostgreSQL 연동을 위해 다음 방식을 제공합니다.

| 방식 | 용도 |
| --- | --- |
| `postgresql()` table function | 일회성 조회, 테스트, 임시 적재 |
| `ENGINE = PostgreSQL(...)` | ClickHouse에 외부 PostgreSQL 테이블처럼 등록 |
| `INSERT INTO ... SELECT FROM postgresql(...)` | WarehousePG/EDB 데이터를 ClickHouse MergeTree 테이블로 복사 |

이 문서에서는 먼저 `postgresql()` table function으로 연결을 검증한 뒤, 필요하면 PostgreSQL table engine 또는 ClickHouse 내부 테이블 적재로 확장합니다.

```text
WarehousePG/EDB
  └─ PostgreSQL protocol :5432/5444
        ↓
ClickHouse ch01
  ├─ postgresql() table function
  ├─ PostgreSQL table engine
  └─ MergeTree / ReplicatedMergeTree 적재
```

## 1. 사전 준비

### WarehousePG/EDB 접속 정보

아래 값은 실제 환경에 맞게 변경합니다.(해당 실습에서는 WarehousePG에서 수행하였습니다.)

| 항목 | 예시 |
| --- | --- |
| Host | `10.0.15.112` |
| Port | `5432` |
| Database | `postgres` |
| Schema | `public` |
| Table | `sales_orders` |
| User | `clickhouse_reader` |
| Password | `data12!@` |

### 네트워크 확인

ClickHouse 노드에서 WarehousePG/EDB의 PostgreSQL port에 접근 가능해야 합니다.

```bash
nc -vz 10.0.15.112 5432
```

`nc`가 없으면 다음처럼 `/dev/tcp`로 간단히 확인할 수 있습니다.

```bash
timeout 3 bash -c '</dev/tcp/10.0.15.112/5432'
echo $?

```

결과가 `0`이면 TCP 연결이 가능합니다.

AWS 환경에서는 Security Group inbound/outbound에서 다음 방향이 열려 있어야 합니다.

```text
ClickHouse node -> WarehousePG/EDB : TCP 5432/5444
```

## 2. WarehousePG/EDB 사용자 및 테스트 테이블

WarehousePG/EDB에 접속하여 ClickHouse 연동 전용 사용자를 생성합니다.

```sql
CREATE USER clickhouse_reader WITH PASSWORD 'data12!@';
```

테스트용 테이블을 생성합니다.

```sql
CREATE TABLE public.sales_orders
(
    order_id    BIGINT PRIMARY KEY,
    customer_id BIGINT,
    status      TEXT,
    amount      NUMERIC(18, 2),
    created_at  TIMESTAMP
);
```

테스트 데이터를 입력합니다.

```sql
INSERT INTO public.sales_orders
    (order_id, customer_id, status, amount, created_at)
VALUES
    (1, 101, 'paid',     12000.00, now() - interval '10 minutes'),
    (2, 102, 'paid',      8300.50, now() - interval '8 minutes'),
    (3, 103, 'cancelled', 2500.00, now() - interval '5 minutes'),
    (4, 104, 'pending',   9900.00, now() - interval '1 minutes');
```

조회 권한을 부여합니다.

```sql
GRANT USAGE ON SCHEMA public TO clickhouse_reader;
GRANT SELECT ON public.sales_orders TO clickhouse_reader;
```


## 3. WarehousePG/EDB 접속 허용 설정

ClickHouse의 `postgresql()` table function은 SSH를 사용하지 않고 PostgreSQL protocol로 직접 접속합니다. 따라서 WarehousePG/EDB 서버에서 ClickHouse 노드가 TCP 5432로 접속할 수 있도록 허용해야 합니다.

이번 테스트에서는 ClickHouse에서 WarehousePG/EDB의 private IP로 접속했습니다.

```text
WarehousePG/EDB endpoint: 10.0.15.112:5432
Database: postgres
Table: public.sales_orders
User: clickhouse_reader
```

### AWS Security Group

AWS 내부 통신은 public IP보다 private IP를 사용하는 것을 권장합니다. WarehousePG/EDB EC2의 Security Group inbound에 ClickHouse 노드 또는 ClickHouse Security Group을 허용합니다.

권장 방식:

```text
Type: PostgreSQL
Protocol: TCP
Port: 5432
Source: ClickHouse EC2 Security Group
```

IP로 직접 허용할 경우 ClickHouse private IP를 `/32`로 추가합니다.

```text
10.0.5.165/32
10.0.9.19/32
10.0.4.51/32
```

### postgresql.conf

WarehousePG/EDB가 외부 접속을 받을 수 있도록 `listen_addresses`를 확인합니다.

```conf
listen_addresses = '*'
```

또는 서버 private IP만 명시합니다.

```conf
listen_addresses = 'localhost,10.0.15.112'
```

`127.0.0.1`에만 listen 중이면 ClickHouse 노드에서 접속할 수 없습니다.

```bash
ss -lntp | grep 5432
```

정상 예시:

```text
0.0.0.0:5432
```

또는:

```text
10.0.15.112:5432
```

### pg_hba.conf

WarehousePG/EDB의 `pg_hba.conf`에 ClickHouse 노드의 private IP를 허용합니다.

EDB/PostgreSQL 예시:

```conf
host    all     clickhouse_reader   10.0.5.165/32      scram-sha-256
host    all     clickhouse_reader   10.0.9.19/32       scram-sha-256
host    all     clickhouse_reader   10.0.4.51/32       scram-sha-256
```

테스트 환경에서 md5를 사용하는 경우:

```conf
host    all     clickhouse_reader   10.0.5.165/32      md5
host    all     clickhouse_reader   10.0.9.19/32       md5
host    all     clickhouse_reader   10.0.4.51/32       md5
```

WHPG 에서 `gpadmin`으로 테스트하는 경우에는 다음처럼 추가할 수 있습니다.

```conf
host    all     gpadmin             10.0.5.165/32      trust
host    all     gpadmin             10.0.9.19/32       trust
host    all     gpadmin             10.0.4.51/32       trust
```

단, `trust`는 비밀번호 없이 접속을 허용하므로 테스트 환경에서만 사용하고 운영 환경에서는 별도 연동 계정과 `md5` 또는 `scram-sha-256` 사용을 권장합니다.

설정 변경 후 reload합니다.

EDB/PostgreSQL:

```bash
pg_ctl reload
```

systemd 서비스인 경우:

```bash
systemctl reload edb-as-18
```

WHPG:

```bash
gpstop -u
```

ClickHouse 노드에서 TCP 연결을 다시 확인합니다.

```bash
nc -vz 10.0.15.112 5432
```

ClickHouse 공식 문서 기준으로 단순 `WHERE` 조건의 일부 비교 연산은 PostgreSQL 쪽에서 실행될 수 있지만, 복잡한 집계, 정렬, 조인 등은 ClickHouse 쪽에서 처리될 수 있습니다. 큰 테이블을 조회할 때는 WarehousePG/EDB 쪽에서 충분히 필터링되도록 주의합니다.

## 5. PostgreSQL table engine으로 외부 테이블 등록

자주 조회할 테이블이면 ClickHouse에 외부 테이블처럼 등록할 수 있습니다.

```sql
CREATE DATABASE IF NOT EXISTS ext;
```

```sql
CREATE TABLE ext.sales_orders_pg
(
    order_id    Int64,
    customer_id Int64,
    status      String,
    amount      Decimal(18, 2),
    created_at  DateTime
)
ENGINE = PostgreSQL(
    '10.0.15.112:5432',
    'postgres',
    'sales_orders',
    'clickhouse_reader',
    'data12!@',
    'public'
);
```

조회:

```sql
SELECT *
FROM ext.sales_orders_pg
ORDER BY order_id;
```

이 방식은 데이터를 ClickHouse에 저장하는 것이 아니라, 조회 시 WarehousePG/EDB에서 데이터를 읽습니다.

```text
ClickHouse ext.sales_orders_pg
  -> WarehousePG/EDB public.sales_orders
```

## 6. ClickHouse 내부 테이블로 적재

분석 성능이 필요하면 WarehousePG/EDB 데이터를 ClickHouse 내부 테이블로 복사합니다.

```sql
CREATE DATABASE IF NOT EXISTS warehouse
ON CLUSTER my_cluster;
```

```sql
CREATE TABLE IF NOT EXISTS warehouse.sales_orders_local
ON CLUSTER my_cluster
(
    order_id    Int64,
    customer_id Int64,
    status      LowCardinality(String),
    amount      Decimal(18, 2),
    created_at  DateTime
)
ENGINE = ReplicatedMergeTree(
    '/clickhouse/tables/{shard}/warehouse/sales_orders_local',
    '{replica}'
)
PARTITION BY toYYYYMM(created_at)
ORDER BY (created_at, order_id);
```

클러스터 조회용 Distributed table을 생성합니다.

```sql
CREATE TABLE IF NOT EXISTS warehouse.sales_orders
ON CLUSTER my_cluster
AS warehouse.sales_orders_local
ENGINE = Distributed(
    my_cluster,
    warehouse,
    sales_orders_local,
    rand()
);
```

초기 데이터를 적재합니다.

> `INSERT SELECT`는 한 노드에서만 실행합니다. 여러 노드에서 동시에 실행하면 같은 데이터가 중복 적재될 수 있습니다.

```sql
INSERT INTO warehouse.sales_orders
SELECT
    order_id,
    customer_id,
    status,
    toDecimal64(amount, 2) AS amount,
    created_at
FROM postgresql(
    '10.0.15.112:5432',
    'postgres',
    'sales_orders',
    'clickhouse_reader',
    'data12!@',
    'public'
);
```

적재 확인:

```sql
SELECT
    count() AS rows,
    min(created_at) AS min_time,
    max(created_at) AS max_time
FROM warehouse.sales_orders;
```

## 7. 증분 적재 예시

운영 환경에서는 전체 데이터를 매번 복사하지 않고, 기준 컬럼으로 증분 적재합니다.

예를 들어 `order_id`가 증가하는 값이라면 현재 ClickHouse에 적재된 최대 `order_id` 이후 데이터만 가져옵니다.

```sql
INSERT INTO warehouse.sales_orders
SELECT
    order_id,
    customer_id,
    status,
    toDecimal64(amount, 2) AS amount,
    created_at
FROM postgresql(
    '10.0.15.112:5432',
    'postgres',
    'sales_orders',
    'clickhouse_reader',
    'data12!@',
    'public'
)
WHERE order_id > (
    SELECT ifNull(max(order_id), 0)
    FROM warehouse.sales_orders
);
```

`updated_at` 같은 변경 시각 컬럼이 있다면 시간 기준 증분 적재도 가능합니다.

```sql
WHERE updated_at > (
    SELECT ifNull(max(updated_at), toDateTime('1970-01-01 00:00:00'))
    FROM warehouse.sales_orders
)
```

단순 증분 방식은 UPDATE/DELETE 반영이 어렵습니다. 변경/삭제까지 동기화해야 하는 경우에는 CDC 도구나 별도의 적재 파이프라인을 검토합니다.

## 8. Named Collection 사용

운영 환경에서는 SQL에 접속 정보를 직접 쓰는 대신 named collection 사용을 권장합니다.

각 ClickHouse 노드에 `/etc/clickhouse-server/config.d/postgresql-named-collections.xml`을 생성합니다.

```xml
<clickhouse>
    <named_collections>
        <warehouse_pg>
            <host>10.0.15.112</host>
            <port>5432</port>
            <database>postgres</database>
            <user>clickhouse_reader</user>
            <password>data12!@</password>
            <schema>public</schema>
        </warehouse_pg>
    </named_collections>
</clickhouse>
```

권한:

```bash
chown root:clickhouse /etc/clickhouse-server/config.d/postgresql-named-collections.xml
chmod 640 /etc/clickhouse-server/config.d/postgresql-named-collections.xml
```

ClickHouse 재시작:

```bash
systemctl restart clickhouse-server
```

사용 예시:

```sql
SELECT *
FROM postgresql(warehouse_pg, table='sales_orders')
LIMIT 10;
Query id: 455b1f83-d1e4-4f49-8ca4-97d0b363248c

   ┌─order_id─┬─customer_id─┬─status────┬─amount─┬─────────────────created_at─┐
1. │        3 │         103 │ cancelled │   2500 │ 2026-07-14 13:01:23.409449 │
2. │        1 │         101 │ paid      │  12000 │ 2026-07-14 12:56:23.409449 │
3. │        2 │         102 │ paid      │ 8300.5 │ 2026-07-14 12:58:23.409449 │
4. │        4 │         104 │ pending   │   9900 │ 2026-07-14 13:05:23.409449 │
   └──────────┴─────────────┴───────────┴────────┴────────────────────────────┘

4 rows in set. Elapsed: 0.068 sec. 
```

> Named collection 파일에는 비밀번호가 들어가므로 파일 권한을 제한합니다.

## 9. 장애 확인 포인트

### TCP 연결 실패

```bash
nc -vz 10.0.15.112 5432
```

확인 대상:

```text
WarehousePG/EDB listen address
pg_hba.conf
AWS Security Group
OS firewall
라우팅/VPC
```

### 인증 실패

WarehousePG/EDB에서 사용자와 비밀번호를 확인합니다.

```sql
SELECT usename
FROM pg_user
WHERE usename = 'clickhouse_reader';
```

### 권한 부족

WarehousePG/EDB에서 schema/table 권한을 확인합니다.

```sql
GRANT USAGE ON SCHEMA public TO clickhouse_reader;
GRANT SELECT ON public.sales_orders TO clickhouse_reader;
```

ClickHouse에서 외부 PostgreSQL 접근 권한을 확인합니다.

```sql
SHOW GRANTS FOR clickhouse_admin;
```

### 타입 변환 오류

PostgreSQL의 `numeric`, `timestamp with time zone`, `jsonb`, `uuid`, 배열 타입 등은 ClickHouse 타입과 정확히 맞춰야 합니다.

먼저 적은 컬럼으로 테스트합니다.

```sql
SELECT
    order_id,
    status,
    created_at
FROM postgresql(
    '10.0.15.112:5432',
    'postgres',
    'sales_orders',
    'clickhouse_reader',
    'data12!@',
    'public'
)
LIMIT 10;
```
