### Keeper 설정 의미

#### listen_host

```xml
<listen_host>0.0.0.0</listen_host>
```

Keeper가 로컬호스트뿐 아니라 다른 노드의 접속도 받도록 합니다.

#### tcp_port

```xml
<tcp_port>9181</tcp_port>
```

ClickHouse Server가 Keeper에 접속하는 포트입니다.

```text
ClickHouse Server → Keeper
```

#### server_id

```xml
<server_id>1</server_id>
```

Keeper 노드 자체의 고유 ID입니다.
반드시 노드별로 달라야 합니다.

```text
ch01 = 1
ch02 = 2
ch03 = 3
```

#### raft_configuration

```xml
<raft_configuration> / </raft_configuration>
```

Keeper 3대가 어떤 노드로 구성되어 있는지 정의합니다.

#### 9234 포트

```xml
<port>9234</port>
```

Keeper끼리 Raft 통신을 하는 포트입니다.
Keeper ↔ Keeper
9181과 용도가 다릅니다.

```text
9181 = ClickHouse Server가 Keeper에 접속
9234 = Keeper 노드끼리 Raft 통신
```

### cluster.xml 의미

#### remote_servers
```xml
<remote_servers>
    <my_cluster>

ClickHouse 서버 클러스터 이름을 my_cluster로 정의합니다.
SQL에서는 다음처럼 사용합니다.
ON CLUSTER my_cluster
```

#### shard
```xml
<shard>

하나의 데이터 분할 단위입니다.
현재는 shard가 하나뿐입니다.
Shard 01
```

#### replica
```xml
<replica>

같은 shard의 데이터 복제본입니다.
현재는 세 개입니다.
Replica 01 = ch01
Replica 02 = ch02
Replica 03 = ch03
```

#### internal_replication
```xml
<internal_replication>true</internal_replication>

Distributed 테이블에 INSERT할 때 같은 shard의 replica 전부에 직접 쓰지 않고, 하나의 replica에 쓰면 ReplicatedMergeTree가 나머지 replica로 복제하도록 합니다.
```

#### zookeeper
```text
설정 이름은 zookeeper지만 실제 연결 대상은 ClickHouse Keeper입니다.
Keeper가 ZooKeeper 프로토콜과 호환되기 때문에 설정 태그 이름이 이렇게 되어 있습니다.
```

#### macros
```xml
<macros>
    <shard>01</shard>
    <replica>01</replica>
</macros>

테이블 생성 시 다음 변수를 실제 값으로 치환합니다.
{shard}
{replica}
예를 들어 ch02에서는:
{shard}   → 01
{replica} → 02
가 됩니다.
```


## 7. OS 권장 설정

다음 설정은 `ch01`, `ch02`, `ch03` 세 노드에 동일하게 적용합니다.

### Transparent Huge Pages 설정

현재 설정을 확인합니다.

```bash
cat /sys/kernel/mm/transparent_hugepage/enabled
```

다음과 같이 `[always]`가 표시되면 Transparent Huge Pages가 모든 프로세스에 항상 적용되는 상태입니다.

```text
[always] madvise never
```

`always`에서는 메모리 페이지 통합 과정이 ClickHouse 쿼리 지연에 영향을 줄 수 있으므로 `madvise`로 변경합니다.

현재 실행 중인 시스템에 즉시 적용합니다.

```bash
echo madvise > /sys/kernel/mm/transparent_hugepage/enabled
```

재부팅 후에도 유지되도록 RHEL 커널 옵션을 추가합니다.

```bash
grubby --update-kernel=ALL \
  --args="transparent_hugepage=madvise"
```

현재 적용 상태를 다시 확인합니다.

```bash
cat /sys/kernel/mm/transparent_hugepage/enabled
```

정상적으로 적용되면 다음과 같이 `[madvise]`가 표시됩니다.

```text
always [madvise] never
```

`grubby`로 추가한 커널 옵션은 다음 재부팅부터 적용됩니다. 위의 `echo` 명령은 재부팅 전 현재 시스템에 즉시 적용하기 위한 설정입니다.

### Delay accounting 설정

Delay accounting은 프로세스별 CPU 및 I/O 대기시간을 수집하는 Linux 커널 기능입니다. 비활성화되어 있어도 ClickHouse 쿼리, 저장 및 복제는 동작하지만 `OSIOWaitMicroseconds`와 같은 진단 지표가 수집되지 않습니다.

현재 설정을 확인합니다.

```bash
sysctl kernel.task_delayacct
```

현재 실행 중인 시스템에 즉시 적용합니다.

```bash
sysctl -w kernel.task_delayacct=1
```

재부팅 후에도 유지되도록 설정 파일을 생성합니다.

```bash
cat > /etc/sysctl.d/99-clickhouse.conf <<'EOF'
kernel.task_delayacct = 1
EOF

sysctl --system
```

적용 상태를 확인합니다.

```bash
sysctl kernel.task_delayacct
```

정상 출력:

```text
kernel.task_delayacct = 1
```

### 세 노드 설정 확인

각 노드에서 다음 값이 동일하게 표시되는지 확인합니다.

```bash
cat /sys/kernel/mm/transparent_hugepage/enabled
sysctl kernel.task_delayacct
```

## 8. Keeper 시작
세 노드 모두 설정 완료 후 Keeper부터 시작합니다.
세 노드에서 거의 동시에 실행:
```bash
systemctl daemon-reload
systemctl enable clickhouse-keeper
systemctl start clickhouse-keeper

상태확인:
systemctl status clickhouse-keeper --no-pager -l

파일로그:
tail -100 /var/log/clickhouse-keeper/clickhouse-keeper.log
tail -100 /var/log/clickhouse-keeper/clickhouse-keeper.err.log

헬스체크:
clickhouse-keeper-client \
  -h 127.0.0.1 \
  -p 9181 \
  -q "ruok"
정상 출력:
imok

Quorum 확인
clickhouse-keeper-client \
  -h 127.0.0.1 \
  -p 9181 \
  -q "mntr" | egrep 'zk_server_state|zk_synced_followers'

리더 :
zk_server_state        leader
zk_synced_followers    2

팔로워:
zk_server_state        follower
```
