# Vector + ClickHouse + Grafana End-to-End 구성

## 0. 구성 목적

이 문서는 웹 서버에서 발생한 access log를 Vector로 수집하고, ClickHouse에 적재한 뒤 Grafana Dashboard에서 조회하는 end-to-end observability 실습입니다.

기존 ClickHouse 3 node cluster는 그대로 유지하고, `ch01`에만 Nginx, Vector, Grafana를 추가로 설치합니다.

```text
Client
  |
  v
Nginx(ch01)
  |
  | /var/log/nginx/access.log
  v
Vector(ch01)
  |
  | HTTP INSERT :8123
  v
ClickHouse Cluster(ch01/ch02/ch03)
  |
  v
Grafana(ch01)
```

## 1. 서버 구성

| 서버 | 구성 |
| --- | --- |
| ch01 | ClickHouse, Nginx, Vector, Grafana |
| ch02 | ClickHouse |
| ch03 | ClickHouse |

ClickHouse 클러스터는 기존과 동일하게 `my_cluster`를 사용합니다.

```text
ch01: 10.0.5.165
ch02: 10.0.9.19
ch03: 10.0.4.51
```

## 2. 방화벽 포트

`ch01`에서 Nginx와 Grafana 접근을 위해 포트를 허용합니다.

```bash
firewall-cmd --permanent --add-port=80/tcp
firewall-cmd --permanent --add-port=3000/tcp
firewall-cmd --reload
```

ClickHouse HTTP interface를 외부에서 직접 접근할 필요가 없다면 `8123`은 기존 ClickHouse 노드 간 통신 및 로컬 접근 용도로만 사용합니다.

## 4. ClickHouse 로그 적재 테이블 생성

관리자 계정으로 ClickHouse에 접속합니다.

```bash
clickhouse-client \
  --host 10.0.5.165 \
  --user clickhouse_admin \
  --password
```

관측 데이터용 DB를 생성합니다.

```sql
CREATE DATABASE IF NOT EXISTS observability
ON CLUSTER my_cluster;
```

Nginx access log 적재용 local table을 생성합니다.

```sql
CREATE TABLE IF NOT EXISTS observability.nginx_access_local
ON CLUSTER my_cluster
(
    event_time DateTime64(3),
    host LowCardinality(String),
    remote_addr String,
    method LowCardinality(String),
    path String,
    status UInt16,
    bytes UInt64,
    user_agent String,
    message String
)
ENGINE = ReplicatedMergeTree(
    '/clickhouse/tables/{shard}/observability/nginx_access_local',
    '{replica}'
)
PARTITION BY toYYYYMM(event_time)
ORDER BY (host, event_time, status);
```

클러스터 전체 조회용 Distributed table을 생성합니다.

```sql
CREATE TABLE IF NOT EXISTS observability.nginx_access
ON CLUSTER my_cluster
AS observability.nginx_access_local
ENGINE = Distributed(
    my_cluster,
    observability,
    nginx_access_local,
    rand()
);
```

Vector 전용 사용자를 생성하고 INSERT 권한을 부여합니다.

```sql
CREATE USER IF NOT EXISTS vector_user
ON CLUSTER my_cluster
IDENTIFIED WITH sha256_password
BY 'ChangeMe_VectorStrongPassword';

GRANT ON CLUSTER my_cluster
INSERT ON observability.nginx_access_local
TO vector_user;
```

`vector_user`는 Vector가 로그를 적재하는 용도이므로 최소 권한만 부여합니다.

## 6. Vector 로그 파일 읽기 권한

Vector가 `/var/log/nginx/access.log`를 읽을 수 있어야 합니다.

먼저 Vector 실행 계정을 확인합니다.

```bash
systemctl cat vector
id vector
```

RHEL 계열에서는 `/var/log/nginx` 또는 로그 파일 권한에 따라 Vector가 바로 읽지 못할 수 있습니다. 단순 실습에서는 아래처럼 `adm` 그룹을 만들거나 기존 로그 읽기 그룹에 Vector 계정을 추가하는 방식으로 처리할 수 있습니다.

먼저 Vector 계정이 로그 파일을 읽을 수 있는지 확인합니다.

```bash
sudo -u vector test -r /var/log/nginx/access.log
echo $?
```

결과가 `0`이면 읽기 가능하므로 추가 권한 작업은 필요하지 않습니다. 예를 들어 아래와 같이 `/var/log/nginx` 디렉터리에 execute 권한이 있고, `access.log`가 `644`이면 Vector가 읽을 수 있습니다.

```text
drwx--x--x. 2 root root /var/log/nginx
-rw-r--r--. 1 root root /var/log/nginx/access.log
```

결과가 `1`이면 전용 로그 읽기 그룹을 만들어 Vector 계정에 부여합니다.

```bash
groupadd -f nginxlog
usermod -aG nginxlog vector
chgrp nginxlog /var/log/nginx
chgrp nginxlog /var/log/nginx/access.log
chmod 750 /var/log/nginx
chmod 640 /var/log/nginx/access.log
```

로그 로테이션 후에도 권한이 유지되도록 `/etc/logrotate.d/nginx`의 `create` 옵션도 확인합니다.

```bash
grep -n "create" /etc/logrotate.d/nginx
```

필요하면 다음과 같이 맞춥니다.

```text
create 0640 nginx nginxlog
```

## 7. Vector 설정

`/etc/vector/vector.toml`을 작성합니다.

`vector.toml`은 Vector의 파이프라인 설정 파일입니다. 이 파일에서 로그를 어디서 읽을지(source), 읽은 로그를 어떻게 가공할지(transform), 최종적으로 어디로 보낼지(sink)를 정의합니다.

이 예제에서는 다음 흐름으로 구성합니다.

```text
/var/log/nginx/access.log
  -> sources.nginx_access
  -> transforms.parse_nginx
  -> sinks.clickhouse_nginx
  -> ClickHouse observability.nginx_access_local
```

주요 설정 항목은 다음과 같습니다.

Vector 설정의 `[sources.nginx_access]`, `[transforms.parse_nginx]`, `[sinks.clickhouse_nginx]`는 다음 구조입니다.

```text
[컴포넌트종류.컴포넌트이름]
```

`sources`, `transforms`, `sinks`는 Vector에서 정해진 고정 영역입니다. 뒤의 `nginx_access`, `parse_nginx`, `clickhouse_nginx`는 사용자가 정하는 이름이므로 목적에 맞게 바꿀 수 있습니다.

단, 이름을 바꾸면 그 이름을 참조하는 `inputs`도 함께 바꿔야 합니다.

```toml
[sources.nginx_access]

[transforms.parse_nginx]
inputs = ["nginx_access"]

[sinks.clickhouse_nginx]
inputs = ["parse_nginx"]
```

예를 들어 source 이름을 `[sources.web_access_log]`로 바꾸면 transform의 입력도 `inputs = ["web_access_log"]`로 바꿔야 합니다.

| 항목 | 설명 | 수정 기준 |
| --- | --- | --- |
| `data_dir` | Vector가 파일 읽기 위치, 버퍼 등 내부 상태를 저장하는 경로 | 기본값 그대로 사용 |
| `[sources.nginx_access]` | Nginx access log를 읽는 입력 설정 | `sources`는 고정, `nginx_access`는 임의 이름 |
| `read_from = "end"` | Vector 시작 이후 새로 추가되는 로그부터 읽음 | 기존 로그까지 읽고 싶으면 초기 테스트 전에 조정 |
| `[transforms.parse_nginx]` | Nginx 로그 한 줄을 ClickHouse 컬럼에 맞게 파싱 | `transforms`는 고정, `parse_nginx`는 임의 이름 |
| `inputs = ["nginx_access"]` | 어느 source 또는 transform의 결과를 입력으로 받을지 지정 | 참조 대상 컴포넌트 이름과 일치해야 함 |
| `get_hostname!()` | 적재 데이터에 서버 hostname을 넣음 | 여러 웹 서버를 수집할 때 구분용으로 사용 |
| `parse_regex(...)` | access log에서 IP, method, path, status, bytes, user_agent 추출 | 로그 포맷과 맞지 않으면 빈 값 또는 `status = 0` 적재 |
| `event_time` | ClickHouse `DateTime64(3)` 컬럼에 넣을 시간 | ClickHouse가 읽을 수 있도록 문자열 포맷으로 변환 |
| `[sinks.clickhouse_nginx]` | 가공된 로그를 ClickHouse로 보내는 출력 설정 | `sinks`는 고정, `clickhouse_nginx`는 임의 이름 |
| `endpoint` | ClickHouse HTTP interface 주소 | 같은 서버의 ClickHouse로 넣으면 `http://127.0.0.1:8123` |
| `database` / `table` | INSERT 대상 DB와 테이블 | 위에서 생성한 `observability.nginx_access_local`과 일치해야 함 |
| `[sinks.clickhouse_nginx.auth]` | ClickHouse 인증 정보 | `vector_user`와 실제 비밀번호로 변경 |
| `[sinks.clickhouse_nginx.buffer]` | ClickHouse 장애 시 Vector가 데이터를 임시 보관하는 설정 | 실습은 기본값 사용, 운영은 disk buffer 권장 |
| `[sinks.clickhouse_nginx.batch]` | ClickHouse로 데이터를 묶어서 보내는 주기 | 실습은 빠른 확인을 위해 짧게 설정 |

아래 설정은 `ch01`의 Nginx 로그를 `ch01`의 ClickHouse HTTP interface로 전송하는 예시입니다.

```toml
data_dir = "/var/lib/vector"

[sources.nginx_access]
type = "file"
include = ["/var/log/nginx/access.log"]
read_from = "end"

[transforms.parse_nginx]
type = "remap"
inputs = ["nginx_access"]
source = '''
.host = get_hostname!()
.message = to_string(.message) ?? ""

parsed, err = parse_regex(.message, r'^(?P<remote_addr>\S+) - \S+ \[(?P<timestamp>[^\]]+)\] "(?P<method>\S+) (?P<path>\S+) [^"]+" (?P<status>\d+) (?P<bytes>\d+) "[^"]*" "(?P<user_agent>[^"]*)"(?: "[^"]*")?$')

if err == null {
  .remote_addr = parsed.remote_addr
  .method = parsed.method
  .path = parsed.path
  .status = to_int!(parsed.status)
  .bytes = to_int!(parsed.bytes)
  .user_agent = parsed.user_agent
} else {
  .remote_addr = ""
  .method = ""
  .path = ""
  .status = 0
  .bytes = 0
  .user_agent = ""
}

.event_time = format_timestamp!(now(), "%Y-%m-%d %H:%M:%S%.3f")
'''

[sinks.clickhouse_nginx]
type = "clickhouse"
inputs = ["parse_nginx"]
endpoint = "http://127.0.0.1:8123"
database = "observability"
table = "nginx_access_local"
format = "json_each_row"
compression = "gzip"
skip_unknown_fields = true

[sinks.clickhouse_nginx.auth]
strategy = "basic"
user = "vector_user"
password = "data12!@"

[sinks.clickhouse_nginx.buffer]
type = "disk"
max_size = 268435488
when_full = "block"

[sinks.clickhouse_nginx.batch]
timeout_secs = 1
```

`password` 값은 앞에서 생성한 `vector_user`의 비밀번호와 일치해야 합니다.

Vector systemd service가 `/etc/vector/vector.toml`을 사용하도록 `/etc/default/vector`에 설정 파일 경로를 추가합니다.

```bash
vi /etc/default/vector
```

```bash
VECTOR_CONFIG=/etc/vector/vector.toml
```

설정 검증 후 Vector를 시작합니다.

```bash
vector validate /etc/vector/vector.toml
systemctl enable --now vector
systemctl status vector --no-pager
```

Vector 로그 확인:

```bash
journalctl -u vector -f
```

## 8. 적재 테스트

`ch01`에서 Nginx에 요청을 발생시킵니다.

```bash
for i in {1..20}; do
  curl -s "http://localhost/test-${i}" > /dev/null
done
```

ClickHouse에서 적재 결과를 확인합니다.

```sql
SELECT
    event_time,
    host,
    remote_addr,
    method,
    path,
    status,
    bytes
FROM observability.nginx_access
ORDER BY event_time DESC
LIMIT 10;
```

분당 요청 수 확인:

```sql
SELECT
    toStartOfMinute(event_time) AS minute,
    count() AS requests
FROM observability.nginx_access
GROUP BY minute
ORDER BY minute DESC
LIMIT 10;
```

상태 코드별 요청 수 확인:

```sql
SELECT
    status,
    count() AS requests
FROM observability.nginx_access
GROUP BY status
ORDER BY requests DESC;
```

## 9. Grafana 설치

`ch01`에서 Grafana repository를 등록합니다.

```bash
dnf install -y curl
curl -fsSL -o gpg.key https://rpm.grafana.com/gpg.key
rpm --import gpg.key
```

`/etc/yum.repos.d/grafana.repo`를 생성합니다.

```ini
[grafana]
name=grafana
baseurl=https://rpm.grafana.com
repo_gpgcheck=1
enabled=1
gpgcheck=1
gpgkey=https://rpm.grafana.com/gpg.key
sslverify=1
sslcacert=/etc/pki/tls/certs/ca-bundle.crt
```

Grafana OSS를 설치합니다.

```bash
dnf install -y grafana
systemctl enable --now grafana-server
systemctl status grafana-server --no-pager
```

브라우저에서 접속합니다.

```text
http://10.0.5.165:3000
```

기본 계정은 최초 접속 시 변경합니다.

```text
ID: admin
PW: admin
```

## 10. Grafana ClickHouse datasource 설정

Grafana에서 ClickHouse datasource plugin을 설치합니다.

```bash
grafana cli plugins install grafana-clickhouse-datasource
systemctl restart grafana-server
```

Grafana UI에서 datasource를 추가합니다.

| 항목 | 값 |
| --- | --- |
| Type | ClickHouse |
| Server address | `127.0.0.1` |
| Server port | `8123` |
| Protocol | HTTP |
| Database | `observability` |
| Username | `clickhouse_admin` 또는 조회 전용 사용자 |
| Password | 지정한 비밀번호 |

운영 환경에서는 Grafana 전용 조회 사용자를 따로 생성하는 것을 권장합니다.

```sql
CREATE USER IF NOT EXISTS grafana_user
ON CLUSTER my_cluster
IDENTIFIED WITH sha256_password
BY 'data12!@';

GRANT ON CLUSTER my_cluster
SELECT ON observability.nginx_access
TO grafana_user;

GRANT ON CLUSTER my_cluster
SELECT ON observability.nginx_access_local
TO grafana_user;
```

## 11. 장애 확인 포인트

### Nginx 로그가 생성되지 않는 경우

```bash
systemctl status nginx --no-pager
curl -I http://localhost/
tail -n 20 /var/log/nginx/access.log
```

### Vector가 로그 파일을 읽지 못하는 경우

```bash
journalctl -u vector -n 100 --no-pager
sudo -u vector test -r /var/log/nginx/access.log
echo $?
```

`0`이면 읽기 가능, `1`이면 권한 문제가 있습니다.

### Vector 설정 오류

```bash
vector validate /etc/vector/vector.toml
```

### ClickHouse 적재 실패

```bash
journalctl -u vector -n 100 --no-pager
clickhouse-client --query "SELECT count() FROM observability.nginx_access"
```

`vector_user` 권한과 ClickHouse HTTP port `8123` 접근 여부를 확인합니다.

```bash
curl -I http://127.0.0.1:8123/ping
```

정상 응답:

```text
HTTP/1.1 200 OK
```

### Grafana에서 조회가 안 되는 경우

```bash
systemctl status grafana-server --no-pager
journalctl -u grafana-server -n 100 --no-pager
```

Grafana datasource에서 host, port, database, username, password를 다시 확인합니다.
