# API Gateway 로그를 ClickHouse에 저장하고 모니터링하는 방법

API Gateway 로그를 ClickHouse에 저장해서 다각도로 모니터링하려면, **로그 수집 → 전송 → 적재 → 집계 → 시각화**의 5단계 파이프라인을 구성하는 것이 정석입니다.

## 전체 흐름

```
API Gateway → 로그 수집기(Vector/Fluent Bit) → Kafka → ClickHouse(Kafka엔진 + MV) → Grafana 대시보드
```

---

## 1단계. API Gateway — 로그 생성

Kong, Apache APISIX, AWS API Gateway 등 대부분의 오픈API GW는 JSON 라인 형태로 액세스 로그를 남깁니다. 최소한 아래 필드는 확보하는 게 좋습니다:

```json
{
  "timestamp": "2026-07-16T10:23:45Z",
  "request_id": "abc-123",
  "client_ip": "203.0.113.10",
  "method": "GET",
  "path": "/v1/orders",
  "api_key": "partner_a",
  "status_code": 200,
  "latency_ms": 42,
  "upstream_service": "order-service",
  "response_size": 1024
}
```

## 2단계. 로그 수집기 — Vector 또는 Fluent Bit

로그 파일(또는 syslog/stdout)을 실시간으로 tail해서 다음 단계로 전달합니다. Vector가 ClickHouse sink를 기본 지원하므로 잘 맞습니다. 파일 기반이면 아래처럼 구성:

```toml
[sources.gw_logs]
type = "file"
include = ["/var/log/apigw/access.log"]

[sinks.kafka_out]
type = "kafka"
inputs = ["gw_logs"]
bootstrap_servers = "kafka:9092"
topic = "apigw-logs"
encoding.codec = "json"
```

## 3단계. Kafka — 왜 중간에 두는가

ClickHouse로 바로 보내지 않고 Kafka를 거치는 이유:

- **버퍼링**: ClickHouse가 순간 부하로 느려지거나 재시작해도 로그 유실 없이 재처리 가능
- **다중 소비자**: 같은 로그를 ClickHouse뿐 아니라 알림 시스템, 실시간 이상탐지 등에도 동시에 흘려보낼 수 있음
- **백프레셔 완충**: 트래픽 폭주 시에도 안정적 처리

소규모/PoC 단계라면 Kafka 없이 Vector → ClickHouse 직결도 가능합니다.

## 4단계. ClickHouse — 적재 + 실시간 집계

### Raw 로그 테이블 (Kafka 엔진으로 직접 소비)

```sql
CREATE TABLE apigw_logs_queue (
    timestamp DateTime64(3),
    request_id String,
    client_ip String,
    method LowCardinality(String),
    path String,
    api_key LowCardinality(String),
    status_code UInt16,
    latency_ms UInt32,
    upstream_service LowCardinality(String),
    response_size UInt32
) ENGINE = Kafka
SETTINGS
    kafka_broker_list = 'kafka:9092',
    kafka_topic_list = 'apigw-logs',
    kafka_group_name = 'clickhouse_consumer',
    kafka_format = 'JSONEachRow';

-- 실제 저장 테이블 (MergeTree)
CREATE TABLE apigw_logs (
    timestamp DateTime64(3),
    request_id String,
    client_ip String,
    method LowCardinality(String),
    path String,
    api_key LowCardinality(String),
    status_code UInt16,
    latency_ms UInt32,
    upstream_service LowCardinality(String),
    response_size UInt32
) ENGINE = MergeTree()
PARTITION BY toYYYYMMDD(timestamp)
ORDER BY (upstream_service, timestamp)
TTL timestamp + INTERVAL 90 DAY;   -- 90일 후 자동 삭제

-- Kafka 큐 → 실제 테이블로 자동 흘려보내는 Materialized View
CREATE MATERIALIZED VIEW apigw_logs_mv TO apigw_logs AS
SELECT * FROM apigw_logs_queue;
```

### 분 단위 사전집계 테이블 (대시보드 응답속도 확보용)

```sql
CREATE TABLE apigw_stats_1m (
    minute DateTime,
    upstream_service LowCardinality(String),
    status_class LowCardinality(String),   -- '2xx','4xx','5xx'
    request_count AggregateFunction(count),
    avg_latency AggregateFunction(avg, UInt32),
    p99_latency AggregateFunction(quantile(0.99), UInt32)
) ENGINE = AggregatingMergeTree()
ORDER BY (upstream_service, status_class, minute);

CREATE MATERIALIZED VIEW apigw_stats_1m_mv TO apigw_stats_1m AS
SELECT
    toStartOfMinute(timestamp) AS minute,
    upstream_service,
    multiIf(status_code < 300, '2xx', status_code < 500, '4xx', '5xx') AS status_class,
    countState() AS request_count,
    avgState(latency_ms) AS avg_latency,
    quantileState(0.99)(latency_ms) AS p99_latency
FROM apigw_logs
GROUP BY minute, upstream_service, status_class;
```

이렇게 하면 대시보드는 원본 로그 대신 이 **사전집계 테이블**을 조회하므로 데이터가 아무리 쌓여도 빠르게 응답합니다.

## 5단계. Grafana — 대시보드 및 다각도 모니터링

Grafana의 ClickHouse 플러그인으로 연결해서 아래 같은 뷰를 만들 수 있습니다:

| 관점 | 쿼리 아이디어 |
|---|---|
| **트래픽 추이** | 분/시간별 요청 수 시계열 (라인 차트) |
| **에러율** | `status_class` 별 비율 (2xx/4xx/5xx 스택 바) |
| **지연시간 분포** | p50/p95/p99 latency 추이 |
| **API 키/파트너별 사용량** | `api_key` 기준 top-N 테이블 |
| **엔드포인트별 핫스팟** | `path` 기준 요청량/에러율 정렬 |
| **특정 클라이언트 이상탐지** | `client_ip` 기준 급증 패턴 |

```sql
-- Grafana 패널 예시: 서비스별 5xx 에러율 추이
SELECT
    minute,
    upstream_service,
    countMerge(request_count) AS total,
    sumIf(countMerge(request_count), status_class = '5xx') AS errors
FROM apigw_stats_1m
WHERE minute >= now() - INTERVAL 1 HOUR
GROUP BY minute, upstream_service
ORDER BY minute;
```

## 규모별 권장 구성

| 규모 | 구성 |
|---|---|
| 소규모/PoC | API GW → Vector → ClickHouse (Kafka 생략) |
| 중대규모 | API GW → Vector → Kafka → ClickHouse (Kafka엔진+MV) |
| 실시간 알림도 필요 | 위 구성 + Kafka에서 별도 컨슈머로 이상탐지/알림 시스템 분기 |
