# ClickHouse DB 접속 및 기초 DDL/DML

## 1. DB 접속 방법

### 로컬 노드 접속

```bash
clickhouse-client
```

### 원격 노드 접속

```bash
clickhouse-client -h 10.0.5.165
```

기본 사용자에 비밀번호를 아직 설정하지 않았기 때문에 비밀번호 없이 접속됩니다.

### 사용자와 데이터베이스를 지정하여 접속

```bash
ex)
clickhouse-client \
  --host 10.0.5.165 \
  --port 9000 \
  --user app_user \
  --password \
  --database tutorial
```

`--password` 뒤에 비밀번호를 직접 작성하지 않으면 접속할 때 비밀번호 입력 프롬프트가 표시됩니다. 셸 히스토리에 비밀번호가 남지 않도록 비밀번호를 명령행에 직접 입력하지 않는 것을 권장합니다.

### SQL 한 번만 실행

```bash
clickhouse-client \
  --host 10.0.5.165 \
  --query "SELECT hostName(), version()"
```

## 2. 접속 상태 확인

접속 후 현재 사용자, 데이터베이스, 접속 노드 및 버전을 확인합니다.

```sql
SELECT
    currentUser(),
    currentDatabase(),
    hostName(),
    version();
```

클러스터의 모든 Replica가 응답하는지 확인합니다.

```sql
SELECT
    hostName(),
    version()
FROM clusterAllReplicas('my_cluster', system.one);
```

## 4. 데이터베이스 생성 및 확인

세 노드에 `tutorial` 데이터베이스를 생성합니다.

```sql
CREATE DATABASE IF NOT EXISTS tutorial
ON CLUSTER my_cluster;
```

생성된 데이터베이스를 확인하고 현재 세션에서 사용할 데이터베이스를 선택합니다.

```sql
SHOW DATABASES;
USE tutorial;
```

## 5. 계정 및 Role 관리

운영 환경에서는 `default` 사용자를 애플리케이션에서 직접 사용하지 않고, 용도별 사용자와 Role을 생성하여 필요한 권한만 부여하는 것을 권장합니다.

> 아래 비밀번호는 예시입니다. 실제 환경에서는 충분히 긴 임의의 비밀번호로 변경합니다.

### 전용 관리자 계정 생성

초기 설치 직후에는 일반적으로 `default` 사용자로 관리 작업을 수행합니다. `default`의 비밀번호나 접속 범위를 제한하기 전에 전용 관리자 계정을 먼저 생성합니다.

```sql
CREATE USER IF NOT EXISTS clickhouse_admin
ON CLUSTER my_cluster
IDENTIFIED WITH sha256_password
BY 'data12!@';

-- ch01, ch02, ch03에서 각각 실행
GRANT CURRENT GRANTS ON *.*
TO clickhouse_admin
WITH GRANT OPTION;
```

`CURRENT GRANTS`는 명령을 실행하는 `default` 사용자가 실제로 보유하고 있으며 다른 사용자에게 부여할 수 있는 권한만 복사합니다. ClickHouse 버전에 새 권한이 추가되어 `default`가 일부 권한을 갖고 있지 않은 경우, `ALL ON *.*`는 권한 부족 오류가 발생할 수 있습니다.

`GRANT CURRENT GRANTS`는 `ON CLUSTER`와 함께 실행할 수 없습니다. 다음과 같이 각 노드의 `default` 사용자로 접속하여 위 명령을 한 번씩 실행합니다.

```bash
clickhouse-client -h 10.0.5.165 --user default
clickhouse-client -h 10.0.9.19  --user default
clickhouse-client -h 10.0.4.51  --user default
```

각 노드에서 권한이 부여됐는지 확인합니다.

```sql
SHOW GRANTS FOR clickhouse_admin;
```

기존 `default` 세션을 종료하지 않은 상태에서 새 터미널을 열어 관리자 계정으로 접속합니다.

```bash
clickhouse-client \
  --host 10.0.5.165 \
  --user clickhouse_admin \
  --password
```

세 노드에 모두 접속할 수 있는지 확인합니다.

```bash
clickhouse-client -h 10.0.5.165 --user clickhouse_admin --password
clickhouse-client -h 10.0.9.19  --user clickhouse_admin --password
clickhouse-client -h 10.0.4.51  --user clickhouse_admin --password
```

관리 권한이 정상인지 확인합니다.

```sql
SELECT currentUser();
SHOW GRANTS;
SHOW USERS;
SHOW ROLES;
```

사용자 생성과 삭제까지 확인하면 관리자 권한을 더 확실하게 검증할 수 있습니다.

```sql
CREATE USER admin_test_user
ON CLUSTER my_cluster
IDENTIFIED WITH sha256_password
BY 'data12!@';

DROP USER admin_test_user
ON CLUSTER my_cluster;
```

> 새 관리자 계정으로 접속 및 관리 작업이 정상적으로 실행되는지 확인하기 전에는 `default` 사용자를 제한하거나 비활성화하지 않습니다. 잘못 제한하면 관리 가능한 계정이 없어질 수 있습니다.

### default 사용자 제한

`default`는 보통 `/etc/clickhouse-server/users.xml`에서 정의되는 설정 파일 기반 사용자입니다. SQL로 생성한 사용자와 달리 `ALTER USER default`가 거부될 수 있으므로, 각 노드의 `/etc/clickhouse-server/users.d/` 설정을 통해 제한합니다.

먼저 강력한 비밀번호의 SHA-256 해시를 생성합니다.

```bash
printf '%s' 'ChangeMe_DefaultStrongPassword' | sha256sum
```

각 노드에 `/etc/clickhouse-server/users.d/default-user.xml`을 생성합니다. 아래 설정은 기존 빈 비밀번호를 제거하고 해시 비밀번호를 적용하며, `default` 사용자의 접속을 로컬호스트로 제한합니다.

```xml
<clickhouse>
    <users>
        <default>
            <password remove="remove"/>
            <password_sha256_hex>SHA256_해시값</password_sha256_hex>

            <networks replace="replace">
                <ip>127.0.0.1</ip>
                <ip>::1</ip>
            </networks>
        </default>
    </users>
</clickhouse>
```

한 번에 모든 노드를 재시작하지 말고 `ch01`, `ch02`, `ch03` 순서로 한 노드씩 적용합니다.

```bash
systemctl restart clickhouse-server
systemctl status clickhouse-server --no-pager -l
```

각 노드에서 재시작 후 관리자 접속을 다시 확인한 다음 다음 노드로 진행합니다.

```bash
clickhouse-client \
  --host 10.0.5.165 \
  --user clickhouse_admin \
  --password \
  --query "SELECT currentUser(), hostName(), version()"
```

로컬에서 `default` 사용자 접속도 확인합니다.

```bash
clickhouse-client \
  --host 127.0.0.1 \
  --user default \
  --password
```

`9000`, `8123` 포트는 관리자 및 애플리케이션 네트워크에서만 접근하도록 방화벽과 Security Group을 제한합니다. Keeper 통신 포트인 `9181`, `9234`와 서버 간 통신 포트인 `9009`는 ClickHouse 노드 사이에서만 허용합니다.

### 읽기·쓰기 Role 생성

```sql
CREATE ROLE IF NOT EXISTS app_rw
ON CLUSTER my_cluster;

GRANT ON CLUSTER my_cluster
    SELECT, INSERT
ON tutorial.*
TO app_rw;
```

### 조회 전용 Role 생성

```sql
CREATE ROLE IF NOT EXISTS app_ro
ON CLUSTER my_cluster;

GRANT ON CLUSTER my_cluster
    SELECT
ON tutorial.*
TO app_ro;
```

### 사용자 생성 및 Role 할당

```sql
CREATE USER IF NOT EXISTS app_user
ON CLUSTER my_cluster
IDENTIFIED WITH sha256_password
BY 'data12!@';

GRANT ON CLUSTER my_cluster
    app_rw
TO app_user;

ALTER USER app_user
ON CLUSTER my_cluster
DEFAULT ROLE app_rw;
```

조회 전용 사용자는 다음과 같이 생성합니다.

```sql
CREATE USER IF NOT EXISTS report_user
ON CLUSTER my_cluster
IDENTIFIED WITH sha256_password
BY 'data12!@';

GRANT ON CLUSTER my_cluster
    app_ro
TO report_user;

ALTER USER report_user
ON CLUSTER my_cluster
DEFAULT ROLE app_ro;
```

생성한 계정으로 접속합니다.

```bash
clickhouse-client \
  --host 10.0.5.165 \
  --user app_user \
  --password \
  --database tutorial
```

## 7. 기초 DDL

DDL은 데이터베이스, 테이블 및 컬럼과 같은 구조를 생성하거나 변경하는 명령입니다.

### 테이블 생성

기초 SQL 실습을 위한 단일 노드 `MergeTree` 테이블을 생성합니다.

```sql
CREATE TABLE IF NOT EXISTS tutorial.basic_events
(
    event_id UInt64,
    user_id UInt64,
    event_type LowCardinality(String),
    created_at DateTime DEFAULT now()
)
ENGINE = MergeTree
ORDER BY (created_at, event_id);
```

테이블 구조를 확인합니다.

```sql
DESCRIBE TABLE tutorial.basic_events;
SHOW CREATE TABLE tutorial.basic_events;
```

### 컬럼 추가 및 삭제

```sql
ALTER TABLE tutorial.basic_events
    ADD COLUMN source String DEFAULT 'unknown';

ALTER TABLE tutorial.basic_events
    DROP COLUMN source;
```

### 데이터 초기화

`TRUNCATE`는 테이블 구조를 유지하고 데이터를 모두 삭제합니다.

```sql
TRUNCATE TABLE tutorial.basic_events;
```

## 8. 기초 DML

DML은 테이블의 데이터를 조회하거나 입력·변경·삭제하는 명령입니다.

> 아래 DDL과 Mutation 예제는 해당 권한이 있는 관리자 계정으로 실행합니다. 앞에서 만든 `app_user`에는 기본적으로 `SELECT`, `INSERT` 권한만 부여되어 있습니다.

### 데이터 입력

```sql
INSERT INTO tutorial.basic_events
    (event_id, user_id, event_type)
VALUES
    (1, 101, 'login'),
    (2, 102, 'purchase'),
    (3, 103, 'logout');
```

### 데이터 조회

```sql
SELECT *
FROM tutorial.basic_events
ORDER BY event_id;
```

조건, 집계 및 정렬을 사용한 조회 예시입니다.

```sql
SELECT
    event_type,
    count() AS event_count
FROM tutorial.basic_events
WHERE created_at >= now() - INTERVAL 1 DAY
GROUP BY event_type
ORDER BY event_count DESC;
```

### 데이터 변경

```sql
ALTER TABLE tutorial.basic_events
UPDATE event_type = 'signin'
WHERE event_id = 1;
```

### 데이터 삭제

```sql
ALTER TABLE tutorial.basic_events
DELETE WHERE event_id = 3;
```

ClickHouse의 `ALTER TABLE ... UPDATE`와 `DELETE`는 데이터 Part를 다시 작성하는 Mutation입니다. 빈번한 단건 수정·삭제보다는 일괄 처리에 적합하며, 큰 테이블에서는 CPU와 디스크 I/O 부하가 발생할 수 있습니다.

Mutation 진행 상태를 확인합니다.

```sql
SELECT
    database,
    table,
    mutation_id,
    command,
    is_done,
    latest_fail_reason
FROM system.mutations
WHERE database = 'tutorial'
ORDER BY create_time DESC;
```

### 실습 테이블 정리

모든 실습을 마친 후 실행합니다. `DROP`은 테이블 구조와 데이터를 함께 삭제합니다.

```sql
DROP TABLE IF EXISTS tutorial.basic_events;
```

## 11. 복제테이블, Distributed 테이블 관련
현재 구성은 아래와 같습니다.
1 Shard × 3 Replica

ch01: Shard 01 / Replica 01
ch02: Shard 01 / Replica 02
ch03: Shard 01 / Replica 03

현재 데이터 흐름이 이렇게 됩니다.

사용자
  │
  ├─ events_dist        ← 분산 테이블: 데이터 저장 안 함
  │      │
  │      ▼
  └─ events             ← 복제 테이블: 실제 데이터 저장
         ├─ ch01에 저장
         ├─ ch02에 복제
         └─ ch03에 복제

### 1. 복제 테이블이란
아래 구문입니다.
```sql
CREATE TABLE events ON CLUSTER my_cluster
(
    event_id UInt32,
    user_id UInt32,
    event_type String,
    created_at DateTime
)
ENGINE = ReplicatedMergeTree(
    '/clickhouse/tables/{shard}/events',
    '{replica}'
)
ORDER BY event_id;
```
이 테이블은 실제로 데이터를 디스크에 저장하는 테이블입니다.

PostgreSQL식으로 생각하면 그냥 실제 물리 테이블인데, 추가로 다른 ClickHouse 노드와 데이터를 복제하는 기능이 붙어 있다고 보면 됩니다.

ON CLUSTER my_cluster
CREATE TABLE events ON CLUSTER my_cluster

이 SQL을 ch01에서 한 번 실행해도 my_cluster에 등록된 모든 서버에서 실행됩니다.

즉 다음 세 노드에 모두 events 테이블이 생성됩니다.

ch01.default.events
ch02.default.events
ch03.default.events

ON CLUSTER가 없으면 SQL을 실행한 현재 노드에만 테이블이 생성됩니다.

컬럼 정의
(
    event_id UInt32,
    user_id UInt32,
    event_type String,
    created_at DateTime
)

일반적인 테이블 컬럼입니다.

ClickHouse 타입	의미
UInt32	음수가 없는 32비트 정수
String	문자열
DateTime	날짜와 시간

ReplicatedMergeTree
ENGINE = ReplicatedMergeTree(...)

여기가 핵심입니다.
MergeTree는 ClickHouse의 대표적인 실제 저장 엔진이고, ReplicatedMergeTree는 여기에 노드 간 복제 기능이 추가된 엔진입니다.
MergeTree
= 로컬 디스크에 실제 저장
ReplicatedMergeTree
= 로컬 디스크에 실제 저장 + 다른 Replica로 복제

Keeper는 이 복제 상태를 조정합니다. Keeper 자체가 테이블 데이터를 보관하는 것은 아닙니다.
ClickHouse 노드
  └─ 실제 데이터 저장
Keeper
  └─ 어떤 replica에 어떤 데이터 조각이 있는지 관리
Keeper 경로
'/clickhouse/tables/{shard}/events'

이것은 Linux 파일시스템 경로가 아닙니다.
즉 다음 경로를 의미하지 않습니다.
/clickhouse/tables/01/events

리눅스 서버 디렉터리에 데이터가 저장된다는 뜻이 아니라, Keeper 내부에서 이 복제 테이블을 관리하기 위한 논리 경로입니다.
현재 각 노드의 매크로는 다음과 같습니다.
ch01: shard=01, replica=01
ch02: shard=01, replica=02
ch03: shard=01, replica=03

그러므로 {shard}는 모든 노드에서 01로 변환됩니다.
/clickhouse/tables/01/events
세 노드 모두 같은 Keeper 경로를 사용하기 때문에 같은 복제 그룹에 속하게 됩니다.
이 경로 아래에서 세 replica를 관리합니다.
/clickhouse/tables/01/events
  └─ replicas
       ├─ 01
       ├─ 02
       └─ 03
Replica 이름
'{replica}'

각 노드의 cluster.xml에 있는 값을 사용합니다.
ch01:
<replica>01</replica>
ch02:
<replica>02</replica>
ch03:
<replica>03</replica>
따라서 동일한 SQL을 세 노드에 실행해도 실제로는 다음처럼 해석됩니다.

ch01
ReplicatedMergeTree(
    '/clickhouse/tables/01/events',
    '01'
)
ch02
ReplicatedMergeTree(
    '/clickhouse/tables/01/events',
    '02'
)
ch03
ReplicatedMergeTree(
    '/clickhouse/tables/01/events',
    '03'
)

Keeper 경로는 같고 replica 이름만 다릅니다. 이것이 하나의 shard에 속한 3개 복제본이 됩니다.
ORDER BY
ORDER BY event_id
PostgreSQL의 단순 출력 정렬과 의미가 다릅니다.
ClickHouse MergeTree 계열에서 ORDER BY는 데이터를 디스크에 어떤 순서와 인덱스 구조로 저장할지 정합니다.

ORDER BY event_id
→ event_id 기준으로 데이터 정렬 및 sparse index 생성
주로 조회 조건에 자주 사용하는 컬럼을 둡니다.

예를 들어 시간 기준 조회가 많으면:
ORDER BY (created_at, event_id)
같이 만들 수 있습니다.
단, ORDER BY event_id가 PostgreSQL의 PK처럼 중복을 막아주는 것은 아닙니다.
```sql
INSERT INTO events VALUES (1, 101, 'login', now());
INSERT INTO events VALUES (1, 101, 'login', now());
```
둘 다 들어갈 수 있습니다.

### 2. 복제 테이블에 직접 INSERT하면
ch01에서 다음을 실행했다고 하겠습니다.
```sql
INSERT INTO events VALUES
(1, 101, 'login', now());
```
흐름은 다음과 같습니다.

ch01.events에 먼저 저장
       │
       ├─ Keeper가 새 데이터 part 정보 관리
       │
       ├─ ch02.events가 같은 part를 가져감
       │
       └─ ch03.events가 같은 part를 가져감

결과적으로 세 노드 모두 같은 데이터를 보유합니다.
ch01.events: 1건
ch02.events: 1건
ch03.events: 1건
현재는 모든 노드가 같은 shard에 속한 replica이기 때문입니다.

### 3. 분산 테이블이란
아래 구문입니다.
```sql
CREATE TABLE events_dist ON CLUSTER my_cluster
(
    event_id UInt32,
    user_id UInt32,
    event_type String,
    created_at DateTime
)
ENGINE = Distributed(
    my_cluster,
    default,
    events,
    rand()
);
```
이 테이블은 보통 데이터를 직접 저장하지 않습니다.
기존에 만든 events 테이블을 대상으로 질의나 INSERT를 전달하는 논리 테이블입니다.

분산 테이블은 복제 테이블로 만든 events를 가져다가 사용
정확히는 events의 데이터를 복사해서 가져오는 게 아니라, events로 질의를 전달하는 라우터입니다.

events_dist
   │
   └─ my_cluster에 있는 default.events로 질의 전달

### 4. Distributed 구문 해석
ENGINE = Distributed(
    my_cluster,
    default,
    events,
    rand()
);

각 인자는 다음 뜻입니다.
Distributed(
    클러스터 이름,
    데이터베이스 이름,
    대상 로컬 테이블 이름,
    샤딩 키
)

현재 구문으로 풀면:
my_cluster
→ cluster.xml에 정의한 클러스터

default
→ 대상 테이블이 존재하는 데이터베이스

events
→ 실제 데이터를 저장하는 로컬/복제 테이블

rand()
→ 여러 shard 중 어느 shard로 보낼지 정하는 기준

즉 다음 의미입니다.

my_cluster의 각 shard에 존재하는
default.events 테이블을 대상으로
조회와 INSERT를 전달한다
왜 events를 지정하는가

먼저 실제 저장 테이블을 만들었습니다.
CREATE TABLE events ...
ENGINE = ReplicatedMergeTree(...)
그다음 분산 테이블이 이 테이블을 참조합니다.

ENGINE = Distributed(
    my_cluster,
    default,
    events,
    rand()
)

여기서 세 번째 인자인 events가 앞에서 만든 실제 저장 테이블입니다.
따라서 생성 순서는 일반적으로 다음입니다.

- 로컬 저장 테이블 생성
- 분산 테이블 생성

반대로 events가 없는 상태에서 events_dist를 만들 수는 있어도, 실제 조회나 INSERT 시 대상 테이블이 없다는 오류가 발생합니다.

실무에서는 실제 저장 테이블에 _local을 붙이는 경우가 많습니다.
events_local
→ 실제 데이터 저장

events
→ 애플리케이션이 사용하는 Distributed 테이블
예를 들면 다음처럼 구성합니다.

실제 저장 테이블
```sql
CREATE TABLE events_local ON CLUSTER my_cluster
(
    event_id UInt32,
    user_id UInt32,
    event_type String,
    created_at DateTime
)
ENGINE = ReplicatedMergeTree(
    '/clickhouse/tables/{shard}/events_local',
    '{replica}'
)
ORDER BY event_id;
```
분산 테이블
```sql
CREATE TABLE events ON CLUSTER my_cluster
(
    event_id UInt32,
    user_id UInt32,
    event_type String,
    created_at DateTime
)
ENGINE = Distributed(
    my_cluster,
    default,
    events_local,
    rand()
);
```
이 구성이 이름만 봐도 더 이해하기 쉽습니다.
events_local = 물리 저장용
events       = 사용자 접근용
현재 아직 테이블을 만들지 않았다면 이 방식으로 만드는 것을 추천합니다.

### 5. 현재 1 Shard × 3 Replica에서의 동작
여기가 중요합니다.
현재 클러스터에는 shard가 하나뿐입니다.

Shard 01
 ├─ ch01 Replica 01
 ├─ ch02 Replica 02
 └─ ch03 Replica 03
따라서 events 또는 events_dist로 INSERT해도 데이터가 다른 shard로 나뉘지 않습니다.

rand()
→ 선택할 shard가 1개뿐임
→ 무조건 Shard 01

그 후 ReplicatedMergeTree가 같은 shard의 나머지 replica에 데이터를 복제합니다.

INSERT INTO events_dist
          │
          ▼
Shard 01의 replica 하나 선택
          │
          ▼
events_local에 저장
          │
          ├─ ch01 복제
          ├─ ch02 복제
          └─ ch03 복제

즉 현재 구성에서는 Distributed 테이블이 있어도 샤딩 분산 효과는 없습니다.

그래도 다음 이유로 만들 수 있습니다.
- 나중에 shard를 늘릴 때 애플리케이션 SQL을 덜 변경하려고
- 사용자는 항상 분산 테이블만 조회하도록 통일하려고
- 특정 replica가 아니라 클러스터 논리 테이블로 접근하려고

### 6. 샤드가 2개일 때 분산 테이블 역할
예를 들어 서버가 4대이고 다음 구성이라면:

Shard 01
 ├─ ch01 Replica 01
 └─ ch02 Replica 02

Shard 02
 ├─ ch03 Replica 01
 └─ ch04 Replica 02

events_dist에 INSERT하면:
event 1 → Shard 01 → ch01/ch02에 복제
event 2 → Shard 02 → ch03/ch04에 복제
event 3 → Shard 01 → ch01/ch02에 복제
event 4 → Shard 02 → ch03/ch04에 복제

조회하면:
```sql
SELECT * FROM events_dist;
```
Distributed 테이블이 두 shard를 모두 조회한 뒤 결과를 합칩니다.
Shard 01 조회 ─┐
               ├─ 결과 합쳐서 반환
Shard 02 조회 ─┘

반면 로컬 테이블을 직접 조회하면 현재 노드가 속한 shard의 데이터만 보입니다.
```sql
SELECT * FROM events_local;
```
